# 병원 탈출 임시 시나리오 — 작업 순서 및 검증 가이드

이 문서는 **공포 병원 탈출 텍스트 게임**의 임시 시나리오를 프로젝트에 올릴 때, 어떤 순서로 지시하면 안전하게 “한 덩어리로 동작하는 플레이”가 되는지 정리한 것입니다. 아래 **시나리오 원안**은 기획 참고용이며, 구현 시에는 **데이터(상태·단서·대사)** 로 분리해 두는 것을 권장합니다.

---

## 1. 시나리오 원안 (참고)

**역할:** 게임마스터(GM). 플레이어는 폐쇄된 병원에서 탈출한다. 심리 공포 톤.

**규칙 요약**

- 현재 위치 기준으로 행동 결과를 서술한다.
- 조사 / 이동 / 대화 입력에 따라 결과를 반환한다.
- 중요한 단서는 기억(기록)한다.
- 미발견 정보는 절대 선제적으로 말하지 않는다.
- 공포 분위기 유지, 과한 점프스케어 지양.
- NPC 앵무새 **폴리**: 힌트, 가끔 이상한 말 반복, 호감도 변화.

**초기 상태**

- 위치: 폐쇄된 병동 **3층 복도**
- 시간: **23:47**
- 목표: (1) 약장 비밀번호 찾기 (2) 3층 출구 찾기
- 조사 가능: 약장, 벽 게시판, 쓰레기통, 복도 끝 문, 천장 환풍구
- 인벤토리: 녹슨 열쇠, 피 묻은 메모
- 폴리 대사(시작): *「약장 비밀번호는 이 방 어딘가에 있어.」*

**출력 형식(내러티브)**

- 분위기 묘사 2~3줄
- 행동 결과
- 새롭게 발견한 단서
- 필요 시 폴리 대사
- 시스템 메시지 톤 금지, 자연스러운 게임 서술

## 2. 현재 프로젝트와의 갭 (요약)

| 영역 | 현재 상태 | 시나리오와 맞추려면 |
|------|------------|---------------------|
| 행동 처리 | `App.tsx`에서 입력 후 고정 멘트만 추가 | 명령 파싱 + 방/오브젝트 상태에 따른 **결과 생성** |
| 화면 오브젝트 | `GameRoom`: 문·테이블·약장·시계 등 | **게시판·쓰레기통·복도 끝 문·환풍구** 등 시나리오 오브젝트와 ID/이름 정합 |
| 단서·목표 | `CluesPanel` 하드코딩 | 게임 상태와 **동기화** (발견 시에만 표시, 목표 완료 반영) |
| NPC | `NPCPanel` 초기 대사·호감도 고정, 플레이어 입력에 **응답 없음** | 대화 규칙 또는 LLM 연동 + **호감도** 갱신 |
| HUD | 시간·장소 문자열 고정 | 상태에서 **위치·시간(필요 시 진행)** 주입 |
| 빠른 행동 | `CommandInput` 버튼이 `onSendMessage` 미연결 | 시나리오에 맞는 문구로 교체하고 **실제 행동 전달** |
| 로그 톤 | `type: 'system'` 시작 멘트 등 | 시나리오상 “시스템 말투” 지양 → 내러티브/플레이어 행동 구분 정책 정리 |

## 3. 권장 작업 순서 (지시서)

아래 순서는 **의존 관계가 적은 것부터** 쌓아, 나중에 AI(GM)를 붙여도 구조가 흔들리지 않게 하기 위한 순서입니다.

### Phase A — 단일 진실 공급원(Single source of truth)

1. **게임 상태 모델 정의**  
   최소 필드 예: `현재_방_id`, `조사완료_오브젝트`, `보유_아이템`, `발견_단서_id[]`, `목표_완료`, `폴리_호감도`, `대화/로그 히스토리`(선택), `잠금·플래그`(약장 열림 등).

2. **콘텐츠를 데이터로 분리**  
   방별로 `이름`, `설명`, `조사 가능 오브젝트 목록`, 각 오브젝트의 `조건`(선행 단서/아이템), `결과 텍스트`(분위기·행동 결과·신규 단서), `폴리 가산/감산 호감` 등을 JSON 또는 TS 상수로 분리.

3. **출력 템플릿 고정**  
   GM 응답을 항상 “분위기 2~3줄 → 행동 결과 → 신규 단서(없으면 생략) → (조건부) 폴리” 구조로 조립하는 **순수 함수**를 만든다. (나중에 LLM이 와도 후처리/검증에 사용 가능.)

### Phase B — 규칙 엔진(비-AI 또는 경량)

4. **명령 파서(임시)**  
   자연어 전체 대신, 1차로는 키워드 매칭(`약장`, `게시판`, `대화`, `이동` 등) 또는 정규화된 `action: investigate target: cabinet` 형태로 파싱. “미발견 정보 금지”는 **데이터에 없는 텍스트를 엔진이 출력하지 않도록** 템플릿만 사용해 보장한다.

5. **상태 전이 함수**  
   입력 한 번 → `다음 상태` + `로그에 쌓일 문장들` + `UI 패치`(단서 추가, 목표 체크)를 한곳에서 반환. React에서는 `useReducer` 또는 작은 스토어(Zustand 등)로 `App`이 구독하게 한다.

### Phase C — UI 연동

6. **`App`을 허브로 통합**  
   `gameMessages`, 단서, 목표, 인벤토리, 호감도를 모두 상태에서 내려준다. `handleSendMessage`는 스텁이 아니라 **엔진 dispatch**를 호출한다.

7. **`GameRoom` ↔ 시나리오 오브젝트**  
   화면 핫스팟 ID를 데이터의 `objectId`와 1:1로 맞추고, 클릭 시 “해당 오브젝트 조사” 행동을 엔진에 보낸다.

8. **`CluesPanel` / `GameHUD` / 진행도**  
   하드코딩 제거 후 상태 기반 렌더. 진행도는 완료된 목표·필수 단서 비율 등 명확한 규칙으로 계산.

9. **`NPCPanel`**  
   초기 대사는 시나리오 문구로 통일. 플레이어 메시지 전송 시: (규칙 기반 응답 먼저) 또는 (추후) LLM 호출 → 동일 템플릿으로 출력 → 호감도 반영.

10. **`CommandInput` 빠른 행동**  
    `onSendMessage` 또는 전용 콜백으로 실제 행동이 실행되게 연결. 시나리오 오브젝트명과 일치하는 버튼으로 교체.

11. **`GameLog` 메시지 타입 정리**  
    “시스템 말투 금지” 정책에 맞게 `narration` 중심으로 시작 멘트를 바꾸거나, `system`을 개발용으로만 쓰는 등 팀 규칙을 문서화.

### Phase D — 품질 및 확장

12. **저장/불러오기(선택)**  
    `localStorage`에 상태 스냅샷 저장하면 임시 시나리오 테스트 반복이 쉬워진다.

13. **AI GM 연동(선택, 나중)**  
    프롬프트에 “현재 방 JSON + 허용된 사실만 말할 것”을 넣고, 엔진이 **허용된 단서/문장만** 골라주는 가드레일을 둔다. 호감도·폴리 이상 발화는 별도 규칙 또는 확률 테이블.

## 4. 검증용 Todo List

실제 구현 후 아래 항목을 **하나씩 체크**하며 “시나리오대로 돌아가는지” 확인합니다.

### 데이터·규칙

- [ ] 초기 상태가 시나리오와 동일한가: **3층 복도**, **23:47**, 목표 2개, 인벤토리 2종.
- [ ] 조사 가능 오브젝트가 시나리오 목록과 **이름·개수·ID**가 일치하는가.
- [ ] 아직 찾지 않은 비밀번호·출구 정보가 **선행 없이** 노출되지 않는가.
- [ ] 단서는 **발견/획득 시에만** 조사 기록 패널에 나타나는가.
- [ ] 목표 1·2가 각각 완료 조건을 만족하면 체크/진행도에 반영되는가.

### 플레이 플로우

- [ ] 텍스트 명령(조사/이동/대화) 입력 시 **매번** 엔진 결과가 로그에 쌓이는가.
- [ ] 화면에서 오브젝트 클릭 시 **동일한 규칙**으로 결과가 나오는가(텍스트와 클릭 이중 경로 일치).
- [ ] 빠른 행동 버튼이 실제로 행동을 트리거하는가(현재는 미연결 가능성 있음).
- [ ] 잘못된/애매한 입력에 대해 **자연스러운 내러티브** 거절 또는 재질문이 있는가(시스템 말투 아님).

### 폴리(NPC)

- [ ] 시작 시 폴리 대사가 시나리오 문장과 **동일하거나 의미 동등**한가.
- [ ] (설계한 경우) 병원 방송 톤·이상한 말 반복이 **빈도/조건**에 맞게 나오는가.
- [ ] 플레이어 행동 또는 대화 선택에 따라 **호감도**가 변하고 UI에 반영되는가.

### 출력 형식·톤

- [ ] 응답이 **분위기 → 결과 → 단서(선택) → 폴리(선택)** 순으로 읽기 좋은가.
- [ ] “게임이 시작되었습니다”, “오류”, “다음 단계로 이동합니다” 같은 **시스템형 문구**가 플레이어에게 보이지 않도록 정리되었는가.

### 회귀·기술

- [ ] `npm run build`(또는 프로젝트 빌드 명령)가 통과하는가.
- [ ] 저장 기능을 넣었다면 새로고침 후 상태가 복원되는가.
